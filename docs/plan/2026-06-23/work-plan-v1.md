## 0. 한 줄 정의

> 이동만 조종하고 무기는 알아서 발사되며, 끝없이 몰려드는 적의 무리를 누비며 경험치 보석을 모아 레벨업·무기 강화·진화로 점점 강해지는, 터미널 위의 실시간 생존 로그라이트(bullet heaven).

원작 *Vampire Survivors*의 핵심 루프를 터미널 ASCII로 이식한다. 원작의 분위기를 흉내내는 것이 아니라, **재미의 메커니즘**을 옮기는 것이 목표다.

## 1. 범위 (Scope)

### 이번 프로젝트가 충실히 재현할 것 (the fun)

- **이동만 조종(move-only)** — 플레이어는 8방향 이동만 한다. 공격은 전부 자동.
- **타이머 기반 자동 무기** — 무기마다 쿨다운/투사체/타게팅 규칙을 가지고 스스로 발사.
- **경험치 → 레벨업 → 업그레이드 선택 → 진화**의 성장 루프.
- **시간에 따라 격해지는 스폰**(difficulty director) — 화면을 뒤덮는 적과 탄막의 손맛.
- **빌드 다양성** — 무기·패시브 조합과 진화로 매 판이 달라지는 경험.

### 초기 범위 밖 (out of scope, MVP 이후/옵션)

- 온라인·랭킹·클라우드 세이브
- 수십 종의 캐릭터/스테이지/도전과제 전체
- 사운드(터미널 비프) — 후순위 옵션
- 영구 메타 진행(골드·언락) — MVP 포함 여부는 §11 결정 항목
- 컷신·서사

### 비-목표 (non-goals)

- 원작의 픽셀 그래픽/연출 재현(터미널이므로 불가·불필요).
- 60 FPS 동작 보장 — 터미널 I/O 한계 내에서 "충분히 부드러운" 체감을 목표(§3.3, §10 Phase 0).

## 2. 왜 뱀서가 재미있는가 — 이식 시 반드시 살릴 4가지

이식 우선순위는 "그래픽"이 아니라 아래 네 가지 감각이다. 최소 기능으로라도 이 넷이 살아있으면 "뱀서답다".

1. **작은 입력 → 큰 출력.** 이동만 하는데 화면은 점점 난장이 된다. "내가 강해지고 있다"는 즉각적 피드백.
2. **짧은 의사결정의 연속.** 레벨업마다 N택 업그레이드. 1초짜리 선택이 빌드를 만든다.
3. **위험/보상 공간 관리.** 적 무리 사이를 비집고 들어가 경험치를 줍는 긴장. 욕심과 생존의 줄타기.
4. **폭발하는 후반.** 진화 무기가 터지며 화면을 청소하는 카타르시스.

설계 결정이 이 넷 중 무엇과 충돌하면, 이 절을 기준으로 판정한다.

## 3. 터미널 이식의 난제 (이 게임이 '포트'인 이유)

원작은 자유 좌표·고프레임·픽셀 충돌을 전제한다. 터미널은 그렇지 않다. 아래가 이 프로젝트의 진짜 엔지니어링 과제다.

### 3.1 좌표계와 종횡비

- 시뮬레이션은 **실수(float) 월드 좌표**에서 돈다(부드러운 이동/투사체). 렌더 시에만 셀로 양자화한다.
- 터미널 셀은 대략 **세로:가로 ≈ 2:1**로 길쭉하다. 그대로 매핑하면 원형 범위·등속 이동이 **세로로 찌그러진다**.
- 대응: 월드→셀 매핑에서 X를 약 2배 압축(또는 Y 보정). 원형 사거리/확산은 **타원으로 보정**하여 체감상 원형이 되게 한다. 이 매핑 상수(종횡비 보정 계수)는 `config/tuning.toml`로 외부화하고, 적용은 `world.py` 한 곳에 모은다(§5.5).

### 3.2 카메라·뷰포트·컬링

- 월드는 화면보다 크다. **플레이어 중심 카메라**로 월드의 일부만 비춘다.
- 매 프레임 **가시 영역 안의 엔티티만** 렌더한다(컬링). 이는 §3.3 처리량 문제와 직결된다.
- 화면 밖 엔티티도 시뮬레이션은 돈다(스폰은 화면 가장자리 바깥 링에서). 단, 너무 먼 엔티티는 캡/디스폰으로 관리.

### 3.3 렌더 처리량 — 가장 큰 미지수 (feasibility gate)

- 병목은 파이썬 연산이 아니라 **터미널 I/O 처리량**일 공산이 크다(특히 SSH/원격). 위치를 지정해 그리는 글리프 수가 한 프레임에 쓸 수 있는 바이트량을 좌우한다.
- 따라서 "엔티티 몇 개, 몇 FPS"를 **추측으로 못 박지 않는다.** Phase 0에서 측정한 "지속 가능한 글리프 수 N"이 이후 설계(뷰포트 크기·엔티티 캡·TPS)를 **거꾸로 규정**한다.
- 완화 기법(측정 결과에 따라 선택): 뷰포트 축소, 엔티티 캡, **diff 렌더**(이전 프레임과 바뀐 셀만 출력), 색 변경 최소화(escape 시퀀스 비용).

### 3.4 입력

- non-blocking 폴링. `term.inkey(timeout=...)`로 짧게 폴링하며 입력이 없어도 시뮬레이션은 전진(snake 패턴, §4).
- 8방향 이동(화살표; WASD 병행 여부 §11). 대각선은 두 키 동시/연속 입력 처리 정책 필요.

### 3.5 표현력 (가독성)

- 플레이어·적·탄막·픽업·위험지대를 **글리프 + 트루컬러**로 구분한다.
- 한 셀에 여러 엔티티가 겹칠 때 **그리기 우선순위**가 필요(아래일수록 위에 덮음): 바닥 < 픽업 < 적 < 탄막 < 플레이어 < HUD/오버레이. **플레이어는 난장 속에서도 항상 보여야 한다.**

### 3.6 깜빡임 없는 렌더

- 전체 화면 clear 금지. `term.home` 후 프레임 문자열을 한 번에 출력, 짧은 줄은 고정폭 패딩으로 잔상 제거(기존 미니게임 `render.py` 규약 계승). diff 렌더 도입 시 셀 단위 커서 이동 출력.

## 4. 게임 루프와 시간 모델

- **고정 시간 간격(fixed timestep) + accumulator.** 시뮬레이션 틱(TPS)과 렌더를 분리한다. 렌더가 느려도 시뮬레이션의 게임 시간은 일정하게 흐르게 한다(프레임 드랍 시 따라잡기/상한 포함).
- 목표 TPS와 렌더 주기는 **Phase 0 측정 후 확정**(현재 잠정: 시뮬레이션 15~20 TPS 가정, 추측값). 확정값은 `config/tuning.toml`로 외부화한다(§5.5).
- 루프 골격은 `snake/main.py`의 검증된 패턴(폴링+timeout, `dirty` 플래그, tick interval)을 기반으로 한다. 의사 코드는 부록 A.
- 상태: 진행/일시정지/게임오버/재시작.

## 5. 시뮬레이션 아키텍처

### 5.1 모듈/파일 구조 (제안, 패키지 레이아웃)

```
terminal_vs/
  __main__.py        # 엔트리: 터미널 셋업 + 메인 루프 (snake/main.py 패턴)
  loop.py            # fixed-timestep 루프, 입력 폴링
  config.py          # 외부 설정 파일 로드·검증 → 불변 설정 객체 (§5.5)
  world.py           # 월드↔셀 좌표 매핑, 종횡비 보정, 카메라/뷰포트
  sim/
    state.py         # 런타임 시뮬레이션 상태(가변 버퍼) — §6 경계의 '가변' 쪽
    step.py          # 한 틱 파이프라인(아래 순서)
    spatial.py       # uniform grid 공간 해시(충돌 질의)
    spawn.py         # difficulty director(시간 기반 스폰)
  rules/             # §6 경계의 '순수·불변' 쪽 — 기존 game.py처럼 테스트 가능
    weapons.py       # 무기 정의·발사 판정·타게팅(순수 함수)
    damage.py        # 데미지/넉백 계산(순수)
    evolution.py     # 진화 자격 판정(순수)
    leveling.py      # 경험치 곡선/레벨업 선택지(순수)
    defs.py          # 밸런스 테이블 — config 파일에서 로드한 불변 데이터
  render/
    frame.py         # 레이어 합성, diff 렌더
    hud.py           # HUD/오버레이
  content/           # MVP 콘텐츠 데이터(후속 문서에서 확장)
config/              # 게임 상수·밸런스 (코드 수정 없이 튜닝, §5.5)
  balance.toml       # 무기/적/진화/레벨링/디렉터 밸런스 상수
  tuning.toml        # 시뮬레이션 TPS·뷰포트·엔티티 캡·폴링 등 운영 상수
tests/ (또는 selftest.py)
run.sh
```

### 5.2 엔티티 모델

- 종류: `player`, `enemies`, `projectiles`, `pickups`, `hazards`(장판/오라).
- 런타임 표현은 **가변 버퍼**(리스트/구조체)로 두고 sim step 안에서 제자리 갱신한다(§6).
- 각 엔티티 공통 필드(예): 위치(x,y float), 속도, HP, 팀, 글리프/색, 수명(투사체).

### 5.3 충돌·공간 분할

- **uniform grid spatial hash**: 월드를 셀 크기 버킷으로 나눠, 엔티티를 버킷에 등록. 충돌 질의는 인접 버킷만 검사(전수 비교 O(n²) 회피).
- 뱀서식 충돌은 대부분 "플레이어 무기 vs 적", "적 vs 플레이어"의 원-원/원-점 근사. 정밀 픽셀 충돌 불필요.

### 5.4 틱 파이프라인(시스템 순서)

```
1) input 적용(플레이어 의도 속도)
2) director 스폰(화면 밖 링에서 적 생성)
3) 적 AI 이동(주로 플레이어 추적)
4) 무기 쿨다운 갱신 → 발사(타게팅 규칙으로 투사체 생성)
5) 투사체 이동/수명
6) 충돌 해소(무기↔적 데미지/넉백, 적↔플레이어 피해)
7) 사망 처리 → 경험치 보석 드랍
8) 픽업 수집(자석 범위) → 경험치 적립 → 레벨업 트리거
9) 정리(죽은 엔티티 제거, 원거리 디스폰)
10) 카메라 갱신
```

레벨업 트리거 시 루프는 **선택 오버레이 모드**로 전환(시뮬레이션 일시정지).

### 5.5 설정·상수 관리 (config)

게임에 영향을 주는 **중요 상수는 코드에 하드코딩하지 않고 외부 설정 파일**(`config/*.toml`)에 모은다. 밸런스 조정과 튜닝을 코드 수정·재배포 없이 파일 편집만으로 할 수 있게 하려는 것이다.

- **밸런스 상수(`balance.toml`):** 무기별 쿨다운·데미지·투사체 수/속도, 적 HP·이동속도·스폰 가중치, 경험치 곡선, 진화 자격 조건, 패시브 효과치(공속·이속·자석 범위), 디렉터 난이도 스텝.
- **튜닝 상수(`tuning.toml`):** 시뮬레이션 TPS, 폴링 타임아웃, 따라잡기 상한, 뷰포트 크기, 엔티티 캡, 종횡비 보정 계수 등 Phase 0 측정으로 확정되는 운영 상수.
- **로딩·검증:** `config.py`가 시작 시 설정 파일을 읽어 스키마·범위를 검증한 뒤 **불변 설정 객체**로 만든다. 누락 키는 코드에 둔 기본값으로 폴백하므로 설정 파일이 없어도 첫 실행이 보장되고, 잘못된 값은 모호하지 않은 에러로 보고한다.
- **불변성·테스트 정합(§6, §13):** 로드된 설정 객체는 불변이며 `rules/defs.py`가 이를 읽어 밸런스 테이블을 구성한다. 규칙 계층은 전역 접근 대신 설정을 **읽기 전용 의존성으로 주입**받아, 헤드리스 테스트에서 임의 설정을 주입해 결정적으로 검증할 수 있다(`random.Random` 주입과 동일한 원칙).
- **형식:** 사람이 읽고 고치기 쉬운 `TOML`을 기본 형식으로 권장한다(Python 3.11+ 표준 라이브러리 `tomllib`로 추가 의존성 없이 파싱). JSON 등 대안은 호환성 요구에 따라 선택 가능하다.

## 6. 불변성 정책 — ADR-001 (전역 규약의 의도적·한정적 예외)

> **장군의 승인이 필요한 항목이다(§11).** 전역 코딩 규약은 불변성을 CRITICAL로 규정한다("ALWAYS create new objects, NEVER mutate"). 실시간 시뮬레이션과 이 규칙은 정면으로 만난다. 이를 숨기지 않고 명시적으로 다룬다.

### 맥락

- 매 틱 수십~수백 엔티티의 위치/HP/타이머가 갱신된다(잠정 규모; Phase 0에서 확정).
- 매 틱 전 월드를 새 객체로 복제하면 코드는 규칙을 지키지만, 불필요한 할당이 늘고 코드도 장황해진다.

### 결정 (경계 분할)

- **순수·불변(규칙):** 데미지 계산, 무기 타게팅, 진화 자격, 레벨업 선택지 산출, 밸런스 테이블은 **부수효과 없는 순수 함수 + 불변 값/설정 타입**으로 둔다. 이로써 게임의 "두뇌"는 기존 미니게임들의 `game.py`처럼 헤드리스로 결정적 테스트가 가능하다.
- **가변·제자리(시뮬레이션):** 런타임 엔티티 버퍼(위치·속도·HP·쿨다운)는 `sim/step.py` **내부에서만** 제자리 갱신한다.
- **경계 규율:** 가변 상태는 sim step 밖으로 누출되지 않는다. 렌더/규칙 계층은 버퍼를 **읽기 전용으로만** 본다.

### 과잉교정 금지

- 반대 방향(struct-of-arrays, numpy 벡터화)으로 **조기 최적화하지 않는다.** 이 규모·TPS에서는 순수 파이썬 리스트의 제자리 갱신으로 충분하며, 할당 churn이 병목이 아니다. 프로파일로 입증되기 전에는 단순함을 유지한다.

### 적용 결과

- 규칙 계층은 전역 불변성 규약을 **그대로** 따른다.
- 시뮬레이션 계층은 성능·명료성을 위해 규약의 **한정적 예외**를 적용하되, 경계를 문서화하고 테스트로 보호한다.

## 7. 콘텐츠 — MVP 최소 세트 (루프 증명용)

> 전체 카탈로그(무기 수십 종·진화 트리·적 도감)는 본 문서가 아니라 **후속 콘텐츠 문서**에서 확장한다. 여기서는 "재미 루프가 도는지" 증명할 최소 세트만 정의한다.

- **무기 3종(예시, §11 충실도 결정에 따라 조정):**
  1. 단검류 — 가장 가까운 적 방향으로 투사체 발사.
  2. 근접 휘두르기 — 플레이어 좌우(진행 방향)로 짧은 범위 타격.
  3. 자동 조준 마법탄 — 임의/최근접 적을 추적.
- **패시브 2~3종:** 공격 속도, 이동 속도, 픽업 회수 범위(자석).
- **진화 1종:** 예) 단검 만렙 + 공격속도 패시브 보유 → 진화 무기(관통/연사 강화).
- **적 2종:** (a) 느린 기본 추적형(다수), (b) 빠른 약체 떼.
- **픽업:** 경험치 보석(필수), 회복/자석(옵션).
- **디렉터:** 경과 시간에 따라 스폰율 상승, 1분 단위 강화 스텝. 보스/엘리트는 후순위.

## 8. 진행 시스템

- **레벨업:** 경험치 곡선(레벨당 필요량 증가). 레벨업 시 **N택 업그레이드 카드**를 일시정지 오버레이로 제시(신규 무기/기존 무기 강화/패시브).
- **무기 강화:** 무기는 레벨이 있고, 만렙에서 진화 자격을 얻는다.
- **진화:** "무기 만렙 + 짝 패시브 보유" 충족 시 진화 선택지 등장(`rules/evolution.py` 순수 판정).
- **메타 진행(영구 업그레이드·골드·언락):** MVP 밖 후보. 포함 여부는 §11.

## 9. 렌더링과 UI

- **레이어 합성**(§3.5 우선순위): 바닥 → 픽업 → 적 → 탄막 → 플레이어 → HUD/오버레이.
- **HUD:** HP 바, 레벨·경험치 바, 생존 타이머, 보유 무기/패시브 아이콘, 킬 수.
- **오버레이:** 레벨업 선택, 일시정지, 게임오버(재시작 안내).

## 10. 단계별 로드맵 (마일스톤)

각 phase는 측정 가능한 exit 기준을 가진다. **Phase 0가 타당성의 관문**이며, 그 측정 결과가 이후 모든 수치를 규정한다.

### Phase 0 — 렌더 처리량 스파이크 (feasibility gate)

- 목표: blessed로 위치를 지정해 그리는 **이동 글리프를 한 틱에 몇 개까지 지속 렌더 가능한지** 실제 터미널에서 측정.
- 산출물: 스트레스 하네스 스크립트, 측정표(글리프 수 N × 달성 프레임율), 권장 뷰포트 크기·엔티티 캡·시뮬레이션 TPS 결정.
- 추가 측정: 전체 프레임 출력 vs **diff 렌더**의 처리량 차이, 색 변경 빈도의 영향.
- **Exit:** "이 게임이 부드럽게 가능한 작동점(N, TPS, 뷰포트)"이 숫자로 확정된다. 불가 판정이면 설계를 축소(뷰포트/엔티티)하거나 접근을 재검토.

### Phase 1 — 코어 루프 수직 슬라이스

- 이동 + 카메라 + 적 1종 스폰 + 무기 1종 자동 발사 + 충돌 + 경험치 + 레벨업 1택.
- **Exit:** "이동하며 적을 자동으로 잡고 경험치로 레벨업한다"가 플레이 가능. 재미의 씨앗 확인.

### Phase 2 — 무기/패시브/진화 + MVP 콘텐츠

- 무기 3종·패시브 2~3종·진화 1종·적 2종·디렉터 난이도 곡선.
- **Exit:** 한 판(잠정 5~10분 축소판)에서 빌드가 갈리고 진화가 발동.

### Phase 3 — 폴리시 & 검증

- HUD/오버레이 완성, 밸런스 1차, 게임오버/재시작, **헤드리스 셀프테스트**(규칙 계층 결정적 테스트).
- **Exit:** selftest exit 0, 임포트 스모크 통과, 한 판 완주 가능.

### Phase 4 — 확장 (옵션)

- 메타 진행, 추가 콘텐츠(무기/적/진화), 보스/엘리트, 사운드.

## 13. 기술 규약 (기존 미니게임에서 계승)

- **언어/스택:** Python 3 + `blessed`.
- **결정적 무작위:** `random.Random` 인스턴스 주입 — 시뮬레이션/스폰/드랍을 테스트에서 결정적으로 재현.
- **렌더/로직 분리:** 규칙 계층은 `blessed`에 의존하지 않는다(헤드리스 테스트 가능). `blessed` 의존은 `render/`·`__main__`에만.
- **상수 외부화:** 게임 밸런스·튜닝에 쓰이는 중요 상수는 하드코딩하지 않고 외부 설정 파일(`config/*.toml`)로 관리한다. `config.py`가 로드·검증해 불변 설정 객체로 주입하므로, 코드 수정·재배포 없이 파일 편집만으로 밸런스를 조정할 수 있다(§5.5).
- **식별자/주석/문자열:** 표준 기술 영어. 한자(중국어 문자) 일절 금지.
- **테스트:** `selftest.py`(또는 `tests/`) 헤드리스 실행, 통과 시 exit 0. UI 루프는 대화형이므로 임포트 스모크 + 규칙 계층 단위 테스트로 검증.

## 부록 A. 게임 루프 의사 코드 (snake 패턴 확장)

```python
# 고정 시간 간격 + accumulator. 시뮬레이션 시간은 렌더 속도와 무관하게 흐른다.
cfg = load_config()             # config/tuning.toml 로드·검증 → 불변 설정 (§5.5)
SIM_DT = 1.0 / cfg.sim_tps      # Phase 0에서 sim_tps 확정 (tuning.toml)
POLL_TIMEOUT = cfg.poll_timeout # 입력 폴링(짧게)

def run(term):
    sim = new_run(rng)          # 가변 시뮬레이션 상태
    cam = Camera()
    acc = 0.0
    last = monotonic()
    mode = "play"               # "play" | "levelup" | "pause" | "gameover"

    while True:
        key = term.inkey(timeout=POLL_TIMEOUT)
        intent = map_key(key)   # 이동 의도 / 메뉴 선택 / quit
        if intent == "quit":
            break

        if mode == "play":
            now = monotonic()
            acc += now - last
            last = now
            # 누적 시간을 고정 틱으로 소비(따라잡기), 폭주 방지 상한
            steps = 0
            while acc >= SIM_DT and steps < cfg.max_catchup:
                step(sim, intent, rng)        # §5.4 파이프라인, 제자리 갱신
                acc -= SIM_DT
                steps += 1
            if sim.level_up_pending:
                mode = "levelup"
            if sim.player.hp <= 0:
                mode = "gameover"
            cam.follow(sim.player)
            render_frame(term, sim, cam)      # 가시 영역만, diff 렌더
        elif mode == "levelup":
            handle_levelup_choice(sim, intent)   # 순수 규칙으로 선택지 적용
            if not sim.level_up_pending:
                mode, last = "play", monotonic()
        # pause / gameover 처리 생략
```
