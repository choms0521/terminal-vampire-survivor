# terminal-vampire-survivor — 향후 추가 스펙 백로그

- 작성일: 2026-07-07
- 문서 종류: 기획 문서(백로그/로드맵). 확정된 상세개발 계획이 아니며, 각 항목은 착수 시점에 별도 상세 계획으로 구체화한다.
- 목적: 현재 구현된 baseline 위에 "추가하면 좋을 스펙"을 카테고리·우선순위별로 정리한다. 이미 구현된 기능은 후보에서 제외한다.

## 1. 현재 구현 baseline (2026-07-07 기준)

추가 스펙과 중복을 피하기 위해 현재 상태를 요약한다.

| 영역 | 구현됨 |
|---|---|
| 렌더 | ASCII + emoji 2모드(emoji가 shipped 기본), no-clear redraw, 셀당 `cell_width` 열·행 폭 불변량, 배경 점 격자 |
| 무기 | 8종: dagger, magic_bolt, swing, dagger_evolved, lance, lance_evolved, nova, orbit |
| 패시브 | 3종: attack_speed, move_speed, magnet |
| 적 | 일반 3종(walker, swarm, brute) + 보스 2종(boss_tank, boss_caster) |
| 진화 | dagger 계열, lance 계열(패시브 조건 + 최대 레벨 → 진화 무기 교체) |
| 디렉터 | 가중 스폰, 증원 스텝, 보스 스폰 스케줄(60초·120초) |
| 레벨업 | 3택 드래프트, XP 곡선 |
| 메타 진행 | 킬당 gold 적립, 영구 업글 2종(swift, fury), 런 간 저장 파일(`meta/save.py`) |
| 조작/모드 | play / levelup / pause(`p`), game-over → 재시작 |
| 결정성 | `random.Random` 주입, `TVS_SEED`로 런 고정 |
| 설정 | TOML 튜닝/밸런스 외부화, `TVS_GLYPH_SET` 런치 오버라이드 |

## 2. 우선순위 기준

- **P1 — 근접(가까운 효과/낮은 위험)**: 기존 구조를 그대로 재사용하는 데이터 위주 또는 소규모 변경. 지금의 emoji 작업 흐름과 잘 이어진다.
- **P2 — 중기**: 새로운 시스템/모드를 추가하지만 범위가 한정적이다.
- **P3 — 장기(큰 범위)**: 새 서브시스템, 저장 스키마 변경, 다중 화면 UI 등 설계 비용이 큰 항목.

각 항목은 개요 / 근거 / 대략 범위 / 위험·의존 / 측정 가능한 완료 조건 순으로 적는다.

## 3. 렌더 · UX

### 3.1 [P1] run.sh emoji/ascii 편의 플래그
- 개요: `./run.sh --emoji`, `./run.sh --ascii`가 `TVS_GLYPH_SET`을 대신 세팅하도록 한다. 환경변수를 몰라도 모드 전환 가능.
- 근거: emoji가 기본이 됐으므로, 미지원 터미널 사용자가 `--ascii` 한 번으로 되돌릴 수 있으면 편하다.
- 범위: `run.sh`만 수정(인자 파싱 후 `export TVS_GLYPH_SET`). 코드 변경 없음.
- 위험·의존: 없음. 알 수 없는 플래그는 그대로 `python -m terminal_vs`로 넘겨 기존 인자와 충돌하지 않게 한다.
- 완료 조건: `./run.sh --ascii`로 실행한 프로세스의 `load_default_config().glyph_set == "ascii"`; 플래그 없이 실행 시 emoji.

### 3.2 [P1] 터미널 emoji 지원 자동 감지 → ascii 폴백
- 개요: 런치 시 터미널/로케일이 emoji 렌더 폭 2를 지원하지 않을 가능성이 높으면 자동으로 ascii로 폴백한다(명시 오버라이드가 없을 때만).
- 근거: emoji 기본화의 유일한 위험은 "폭-2로 안 그려지는 터미널"이다. 자동 감지가 있으면 기본화의 위험이 크게 줄어든다.
- 범위: `__main__`에서 감지 로직(예: `TERM`, `LANG`/`LC_*` UTF-8 여부, `sys.stdout.encoding`). 확실하지 않으면 emoji 유지, 명백히 비-UTF-8이면 ascii.
- 위험·의존: 오탐(잘 되는데 ascii로 폴백)이 오히려 나쁠 수 있어 보수적으로 설계. 렌더 로직과 무관(선택만).
- 완료 조건: 비-UTF-8 로케일 모킹 시 override 없이 `glyph_set == "ascii"`; UTF-8에서는 emoji 유지. 명시 `TVS_GLYPH_SET`는 항상 감지를 무시.

### 3.3 [P2] 런타임 glyph_set 토글 핫키
- 개요: 게임 중 키(예: `g`)로 ascii ↔ emoji를 즉시 전환한다.
- 근거: 재실행 없이 비교/폴백 가능. emoji 폭 문제를 사용자가 실시간으로 회피.
- 범위: `cfg`가 frozen이므로 토글 시 `cell_width`/`viewport_w`/`aspect_x`가 재유도돼야 한다 → 런타임에 새 Config를 만들어 교체하는 경로 필요. 카메라·월드 좌표는 논리 셀 기준이라 재계산 필요(반토막/두배 되는 격자).
- 위험·의존: Config 불변성 정책(ADR-001)과 충돌하지 않도록 "교체"로 구현. 전환 프레임에서 no-clear redraw 잔상 처리 주의(전환 시 1회 clear 허용).
- 완료 조건: 토글 후 모든 행 폭이 새 `render_cols`와 일치; 전환 전후 플레이어 월드 좌표 동일.

### 3.4 [P2] 데미지 숫자 · 히트 피드백
- 개요: 적이 피격될 때 짧게 뜨는 부동 데미지 텍스트(또는 색 플래시). 기존 `Effect`(ttl 기반 시각 마커) 재사용.
- 근거: 타격감이 크게 올라간다. Vampire Survivors 계열의 핵심 피드백.
- 범위: 데미지 계산 지점에서 `Effect` 스폰(숫자 문자열 glyph). 렌더는 기존 효과 경로. emoji 모드에서는 숫자도 폭 관리 필요(여러 자리 → 여러 셀).
- 위험·의존: 다자리 숫자의 셀 배치(한 셀에 한 글자) 규칙 필요. 성능(대량 스폰) → ttl 짧게, 상한.
- 완료 조건: 피격 1회당 Effect 1개 생성/만료; emoji 모드 프레임 폭 불변량 유지; 초당 N개 상한 검증.

### 3.5 [P2] 화면 밖 적/보스 방향 표시(엣지 인디케이터)
- 개요: 뷰포트 밖의 보스나 대군 방향을 화면 가장자리에 화살표/마커로 표시.
- 근거: 보스가 화면 밖에서 접근할 때 정보 부족. 생존 판단에 유용.
- 범위: 카메라 밖 엔티티를 가장자리 셀에 투영. 렌더 레이어 추가.
- 위험·의존: HUD/게임플레이 레이어 우선순위 정리. 과밀 시 마커 병합.
- 완료 조건: 보스가 뷰포트 밖일 때 정확한 8방위 셀에 마커; 뷰포트 안이면 사라짐.

### 3.6 [P3] HUD 확장 — 무기 슬롯 · 쿨다운 바 · DPS
- 개요: 보유 무기/패시브 아이콘, 쿨다운 진행 바, 대략 DPS/처치수 등을 HUD에 추가.
- 근거: 빌드 상태 가시성. 현재 HUD는 5요소로 제한적.
- 범위: `render/hud.py` 확장, 레이아웃 재설계. emoji 아이콘 재사용 가능.
- 위험·의존: 세로 공간 예산. 좁은 터미널 대응(축약 모드).
- 완료 조건: 무기 획득 시 슬롯 반영; 쿨다운 바가 실제 쿨다운과 동기; 좁은 폭에서 레이아웃 깨지지 않음.

## 4. 게임 콘텐츠 (데이터 위주 → 대부분 P1)

### 4.1 [P1] 무기 추가
- 개요: 기존 타게팅 전략(nearest / radial / orbit / forward_arc / nearest_or_random)을 재사용하는 신규 무기.
- 후보: 부메랑(왕복 궤적, 신규 이동 필요 시 P2), 체인 라이트닝(연쇄, 신규 타게팅 필요 시 P2), 지속 장판(장판 ttl + 범위 데미지), 후방 발사(nearest의 반대 방향).
- 근거: 빌드 다양성. 데이터만으로 되는 것부터.
- 범위: 데이터만 되는 무기는 `balance.toml` + emoji 매핑 1줄. 신규 이동/타게팅은 sim 로직 추가(P2로 승격).
- 위험·의존: emoji 폭-2 단일 부호 규칙(신규 glyph마다). 밸런스 검증.
- 완료 조건: 신규 무기 스폰/피해/쿨다운이 테스트로 검증; emoji 매핑 시 폭-2 가드 테스트 통과.

### 4.2 [P1] 패시브 추가
- 개요: 신규 스탯 배수 패시브(예: 최대 체력, 피해량, 투사체 관통, 픽업 경험치, 재생).
- 근거: 현재 3종은 빈약. 빌드 선택지 확대.
- 범위: `PassiveDef` + effective_stats 반영 지점. 일부는 스탯 필드 신설 필요(피해 배수, 관통 등).
- 위험·의존: effective_stats 파이프라인에 새 배수 연결. 진화 조건과의 상호작용.
- 완료 조건: 각 패시브 레벨당 스탯 배수 곱 적용을 단위 테스트로 검증.

### 4.3 [P1] 일반 적 다양화
- 개요: 원거리 적(투사체 발사, 보스 caster 로직 재사용), 돌진 적, 분열 적(사망 시 소형 2마리), 실드 적.
- 근거: 현재 일반 적 3종은 이동 속도/HP 차이뿐. 행동 다양성 필요.
- 범위: 원거리는 caster의 `fire_*` 필드 재사용(데이터). 분열/돌진은 sim 로직 추가(P2 경향).
- 위험·의존: 스폰 가중치 재밸런스. 분열은 entity_cap 압박.
- 완료 조건: 신규 적 행동을 결정적 테스트(고정 seed)로 검증; entity_cap 초과 없음.

### 4.4 [P2] 보스 · 엘리트 확장
- 개요: 신규 보스(소환형, 페이즈 전환형)와 엘리트(일반 적의 강화 변형, 특수 드롭).
- 근거: 후반 목표/긴장감. 현재 보스 2종은 60·120초 고정.
- 범위: 보스 스폰 스케줄 확장, 페이즈는 HP 임계 기반 상태 전환 로직 신설.
- 위험·의존: 디렉터 스케줄/보스 풀 가중치. 페이즈 전환은 신규 상태 머신.
- 완료 조건: 보스 스폰 시각/페이즈 전환 임계가 테스트로 검증; 보상 gem/gold 지급 확인.

### 4.5 [P2] 진화 조합 확장
- 개요: 무기×패시브 진화 쌍을 늘린다(예: nova×attack_speed, orbit×magnet).
- 근거: 진화는 빌드 목표. 현재 dagger/lance 2종.
- 범위: `EvolutionDef` 데이터 + 진화 결과 무기 추가. 로직 재사용.
- 위험·의존: 결과 무기 밸런스. 드래프트에서 진화 무기 노출 규칙.
- 완료 조건: 진화 조건 충족 시 무기 교체를 결정적 테스트로 검증.

## 5. 게임 시스템

### 5.1 [P1] 레벨업 드래프트 리롤 · 스킵 · 밴
- 개요: 드래프트 화면에서 리롤(다시 뽑기), 스킵(경험치/소액 보상), 밴(해당 선택지 영구 제외) 지원. VS 계열 표준.
- 근거: 원하는 빌드 유도 가능. 현재는 3택 강제.
- 범위: 레벨업 모드 입력 처리 + 드래프트 재생성. 리롤 횟수는 스탯/자원으로 제한.
- 위험·의존: 결정성 유지(리롤도 주입 RNG). 밴은 런 상태에 목록 보관.
- 완료 조건: 고정 seed에서 리롤 결과 재현; 밴한 항목이 이후 드래프트에 안 나옴.

### 5.2 [P2] 보물상자 · 랜덤 보상
- 개요: 특정 적/보스 처치 시 상자 드롭 → 획득 시 다중 레벨업 또는 골드/특수 보상.
- 근거: VS 계열의 페이싱/도파민 장치.
- 범위: 신규 픽업 타입(`Pickup` 확장 또는 신규 엔티티) + 보상 롤 로직.
- 위험·의존: 보상 테이블 데이터화. 저장 스키마 무관(런 내 소비).
- 완료 조건: 상자 획득 시 보상 롤이 결정적; 다중 레벨업 큐 정상 처리.

### 5.3 [P2] 상태이상 (빙결 · 화상 · 넉백 · 슬로우)
- 개요: 무기/패시브가 적에게 상태이상 부여. 화상=지속 피해, 빙결/슬로우=이동 배수, 넉백=위치 밀림.
- 근거: 전투 깊이. 신규 무기/진화의 개성 부여.
- 범위: `Enemy`에 상태 필드 + step에서 틱 처리. 데이터로 무기에 부여 확률/강도 지정.
- 위험·의존: sim 복잡도 증가. 결정성(틱 순서). 불변성 정책과의 정합.
- 완료 조건: 각 상태의 효과/지속시간을 고정 seed 테스트로 검증.

### 5.4 [P2] 승리 조건 · 런 목표
- 개요: 무한 생존 외에 "N분 생존 시 승리", "최종 보스 처치 승리" 등 명시적 종료.
- 근거: 현재 런은 사망으로만 끝난다. 목표가 있으면 세션 완결감이 생긴다.
- 범위: 디렉터에 종료 조건 추가 + 승리 오버레이. game-over 경로 재사용.
- 위험·의존: 밸런스(도달 가능한 목표). 메타 보상과 연계 여부.
- 완료 조건: 목표 도달 시 승리 상태 전이; 승리/패배가 각각 별도 결과 화면.

### 5.5 [P3] 다중 스테이지 · 맵 변주
- 개요: 스테이지별 배경 격자/스폰 테이블/보스 세트 차별화, 지형 요소(장애물).
- 근거: 반복성 완화. 장기 콘텐츠.
- 범위: 스테이지 정의 데이터 + 스테이지 선택 흐름. 장애물은 충돌/경로 로직 신설.
- 위험·의존: 큰 범위. 월드/충돌 계층 확장.
- 완료 조건: 스테이지별 스폰/보스 차이가 데이터로 구성; 장애물 충돌 테스트.

## 6. 메타 진행

### 6.1 [P1] 영구 업글 확장
- 개요: 현재 2종(swift, fury) 외에 최대 체력, 시작 무기 선택, 픽업 범위, gold 획득량 등 영구 업글 추가.
- 근거: 런 간 성장 동기. 저장/구매 파이프라인 이미 존재(`meta/`).
- 범위: `MetaUpgradeDef` 데이터 + 적용 지점. 스키마 하위호환 유지.
- 위험·의존: 저장 스키마 버전 관리(신규 키 default). 밸런스(가격 곡선).
- 완료 조건: 신규 업글 구매/적용/저장·로드 왕복을 테스트로 검증; 구버전 세이브 로드 시 default 채움.

### 6.2 [P2] 캐릭터 선택 · 해금
- 개요: 시작 스탯/무기가 다른 캐릭터 선택, 조건 달성 시 해금.
- 근거: 리플레이성. 메타 목표.
- 범위: 캐릭터 정의 데이터 + 선택 화면 + 해금 조건 저장.
- 위험·의존: 저장 스키마 확장. 밸런스.
- 완료 조건: 캐릭터별 시작 상태 차이 검증; 해금 상태 저장·로드.

### 6.3 [P2] 도전과제 · 통계
- 개요: 누적 통계(총 킬, 최장 생존, 최고 레벨)와 도전과제. 해금 트리거로도 사용.
- 근거: 장기 동기. 통계는 저장 파일에 누적.
- 범위: 통계 집계 지점 + 저장 스키마 필드 + 표시 화면.
- 위험·의존: 저장 스키마 확장. 집계 성능.
- 완료 조건: 런 종료 시 통계 누적·저장; 도전과제 트리거 검증.

## 7. 인프라 · 품질

### 7.1 [P1] emoji 로스터 폭-2 가드 유지 + 확장 규약 문서화
- 개요: 신규 glyph 추가 시 지켜야 할 규약(단일 부호 폭-2, VS16 금지, `0x1F3xx–0x1F5xx` 선호)을 README/CONTRIBUTING에 명문화. 가드 테스트(`test_all_emoji_glyphs_are_width_two`)는 이미 존재.
- 근거: emoji가 기본이 된 만큼, 잘못된 glyph 추가가 행 폭 불변량을 깨는 회귀를 예방.
- 범위: 문서 1개 섹션. 코드 변경 없음.
- 완료 조건: 규약 문서 존재; 신규 기여 시 참조 가능.

### 7.2 [P2] 리플레이 기록 · 재생
- 개요: `TVS_SEED` + 입력 로그를 기록해 런을 재현/재생. 결정적 sim이므로 seed+입력이면 완전 재현.
- 근거: 버그 재현, 데모, 밸런스 분석에 유용. 결정성 설계를 활용.
- 범위: 입력 타임라인 직렬화 + 재생 모드. 렌더 무관(sim 재현).
- 위험·의존: sim 결정성 완전성 검증(부동소수 재현성 포함). 저장 포맷.
- 완료 조건: 동일 seed+입력 재생 시 최종 상태 해시 일치.

### 7.3 [P2] 밸런스 외부화 마무리 · 난이도 프리셋
- 개요: 남아있는 하드코딩 상수의 TOML 외부화 완료, 난이도 프리셋(easy/normal/hard) 제공.
- 근거: 튜닝 반복 속도. 프로젝트 계획의 밸런스 외부화 목표.
- 범위: 상수 감사 → TOML 이전, 프리셋 파일 선택 로직.
- 위험·의존: no-hardcode 게이트와 정합. 프리셋 전환 UI.
- 완료 조건: 지정 상수의 하드코딩 0건(게이트); 프리셋 전환 시 디렉터/밸런스 값 반영.

### 7.4 [P3] 성능 · 대군 스트레스 대응
- 개요: entity_cap 상향 시 스폰/충돌/렌더 병목 프로파일링과 최적화(공간 분할, 렌더 diff 모드 활용).
- 근거: 후반 대군 페이싱. `render_mode = "diff"` 등 기존 훅 활용 여지.
- 범위: 프로파일 → 병목별 최적화. 벤치(`bench/`) 확장.
- 위험·의존: 결정성 유지하며 최적화. 광범위.
- 완료 조건: 목표 entity 수에서 프레임 예산 내 유지(측정치 명시).

## 8. 추천 진행 순서 (소견)

1. **3.1 run.sh 플래그** + **3.2 emoji 자동 폴백** — 방금 emoji 기본화의 안전망. 위험 최저, 흐름 연속.
2. **5.1 드래프트 리롤/스킵/밴** — 체감 큰 시스템, 결정성만 지키면 범위 한정.
3. **4.1/4.2/4.3 콘텐츠(데이터 위주)** — 빌드 다양성, 대부분 데이터+테스트.
4. **3.4 데미지 숫자** — 타격감. 기존 Effect 재사용.
5. **5.4 승리 조건** — 런 완결감.
6. 이후 P2/P3는 위 결과와 사용자 피드백을 보고 재정렬.

## 9. 비고

- 이 문서는 살아있는 백로그다. 각 항목 착수 시 `docs/development/<착수일>/`에 상세개발 계획으로 구체화하고, 여기서는 상태(착수/완료/보류)만 갱신한다.
- emoji 관련 신규 작업은 반드시 폭-2 단일 부호 규칙과 `test_all_emoji_glyphs_are_width_two` 가드를 준수한다.
- 모든 신규 시스템은 결정성(주입 RNG)·렌더/규칙 분리·Config 불변성(ADR-001)을 유지한다.
